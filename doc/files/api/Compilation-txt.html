Windows

The 3DTouchPad SDK installation provides an API implementation and a few Windows and Linux applications (App) as source code together with precompiled binaries.

It is possible to start a new application directly based on those binaries.

Nevertheless here are instructions for building the API and the Apps.

Building the API

As the installation directory is normally write protected it is recommended to copy api/include and api/src to a separate development folder before building.

The api/src directory contains an hmi-api.sln solution file that could be used with Microsoft Visual C++ 2010.

Depending on the configuration this project builds and standalone mchp_hmi.lib or a combo of the mchp_hmi.dll and an associated mchp_hmi.lib import library.

Building the Apps

In order to build an App the directory of the App has to be copied to a development directory where it could be built.  Each App has its own solution file that could be used with Microsoft Visual C++ 2010.

The Apps build and link against the prebuild binaries in the installation directory.

Linux

This package also contains the source code for building the SDK on Linux.  This requires the content of the following folders:

  • apps/Linux
  • api/include
  • api/src

The directory apps/Linux contains a Makefile that build the API and the Linux apps.  This Makefile expects gcc as the compiler and ar for creating lib-files.  The console-application also requires a curses-compatible library to be installed.

On Ubuntu 14.04 the required packages could be installed with:

sudo apt-get install build-essential libncurses5-dev

Once all requirements are installed the project could be build by going the the apps/Linux directory and running make there.

cd apps/Linux
make

The Makefile currently doesn’t contain instructions for installation.  However the final binaries are located in apps/Linux/build/bin after building finished successfully.

3DTouchPad

The HMI API uses hidapi to connect to the 3DTouchPad.  On Linux there are two implementations of hidapi that are both included in the API source at api/src/io/hidapi/linux.  The one that is used per default is hid.c which in turn uses the hidraw-driver and udev for locating the correct device-file for which it required libudev.  The alternative is hid-libusb.c which uses libusb.  Please read the hidraw readme at api/src/io/hidapi/linux/README.txt for the advantages and shortcomings of the two implementation.

In order to use the libusb implementation replace all occurrences of hid.c in the Makefile with hid-libusb.c and install libusb as stated in the hidraw README.txt.

By default only root has access to the 3DTouchPad device files.  To privilege normal users to interact with the device a udev rule is required.  The following rules change the group for the devices to plugdev and also gives all users read and write access.  It also creates symbolic links for the device at /dev/hmi2d (2D device) and /dev/hmi2dupdate (2D update device) for convenience.

SUBSYSTEM=="usb", ATTRS{idVendor}=="04d8", ATTRS{idProduct}=="09d3", SYMLINK+="hmi2d", GROUP="plugdev", MODE="0666"
SUBSYSTEM=="usb", ATTRS{idVendor}=="04d8", ATTRS{idProduct}=="09d4", SYMLINK+="hmi2dupdate", GROUP="plugdev", MODE="0666"

This rules could be installed by typing the following commands into a terminal window

sudo sh -c 'echo "SUBSYSTEMS==\"usb\", ATTRS{idVendor}==\"04d8\", ATTRS{idProduct}==\"09d3\", SYMLINK+=\"hmi2d\", GROUP=\"plugdev\", MODE=\"0666\"" >> /etc/udev/rules.d/99-hmi.rules'
sudo sh -c 'echo "SUBSYSTEMS==\"usb\", ATTRS{idVendor}==\"04d8\", ATTRS{idProduct}==\"09d4\", SYMLINK+=\"hmi2dupdate\", GROUP=\"plugdev\", MODE=\"0666\"" >> /etc/udev/rules.d/99-hmi.rules'

The new udev rules could either be loaded by a system restart or with the following commands:

sudo udevadm control --reload-rules
sudo udevadm trigger --attr-match=idVendor="04d8" --attr-match=idProduct="09d3"
sudo udevadm trigger --attr-match=idVendor="04d8" --attr-match=idProduct="09d4"

I²C to USB Serial Bridge

The HMI API expects the I2C to USB Serial Bridge (CDC Serial) to be the device at /dev/ttyACM0.  This behavior could be changed in the following line in api/src/io/cdcserial_linux.c:

#define DEVICE "/dev/ttyACM0"

By default only root and users in the dialout group have access to I2C to USB Serial Bridge’s device file.  To privilege normal users to interact with the device a udev rule is required.  The following rule changes the group for this device to plugdev and also gives all users read and write access.  It also creates a symbolic link for the device at /dev/hmi3d for convenience.

SUBSYSTEMS=="usb", ATTRS{idVendor}=="04d8", ATTRS{idProduct}=="000a", SYMLINK+="hmi3d", GROUP="plugdev", MODE="0666"

This rule could be installed by typing the following command into a terminal window

sudo sh -c 'echo "SUBSYSTEMS==\"usb\", ATTRS{idVendor}==\"04d8\", ATTRS{idProduct}==\"000a\", SYMLINK+=\"hmi3d", GROUP=\"plugdev\", MODE=\"0666\"" >> /etc/udev/rules.d/99-hmi.rules'

The new udev rule could either be loaded by a system restart or with the following commands:

sudo udevadm control --reload-rules
sudo udevadm trigger --attr-match=idVendor="04d8" --attr-match=idProduct="000a"
